# 치명적 실수 기록

## 1. MDN Loss에 clamp(min=0.0) 적용

**기간**: epoch 60 ~ 141 (약 80 에포크 낭비)

**원인**: 학습 중 loss가 음수로 나오자 "loss가 음수면 안 된다"고 판단해서 `torch.clamp(loss, min=0.0)` 적용

**왜 틀렸나**: MDN은 연속 확률분포를 모델링한다. 연속분포의 확률밀도는 1을 넘을 수 있고, 따라서 log-likelihood가 양수 → NLL이 음수가 되는 건 **정상**이다. 이산분포와 혼동하면 안 된다.

**결과**: 모델이 잘 예측하는 포인트(음수 loss)의 gradient가 전부 0이 됨 → attention 같은 세부 구조 학습 불가 → 141 에포크 돌려도 텍스트 무시하는 모델 완성

**교훈**: 수학적으로 이해 안 되는 값이 나오면 clamp로 때우지 말고 원리부터 확인해라

---

## 2. Sigma 상한을 2.0으로 제한

**원인**: 생성 결과가 스프링처럼 나오자 sigma가 너무 크다고 판단해서 `clamp(max=2.0)` 적용

**왜 틀렸나**: 학습 초기에 sigma가 큰 건 정상이다. 모델이 아직 확신이 없으니 넓은 분포로 시작하고, 학습이 진행되면서 자연스럽게 줄어든다. 강제로 제한하면 모델의 표현력을 잘라버린다.

**교훈**: 증상을 억지로 누르지 말고 원인을 찾아라. 스프링 출력의 진짜 원인은 attention 미학습이었다.

---

## 3. EOS loss 가중치 2.0

**원인**: end-of-stroke 신호가 중요하다고 판단해서 `stroke_nll + 2.0 * eos_bce`로 설정

**왜 틀렸나**: stroke 좌표 예측이 핵심인데, EOS에 2배 가중치를 주면 모델이 "어디서 끊을지"에만 집중하고 "어떤 좌표를 그릴지"는 상대적으로 소홀해진다.

**교훈**: loss 항목 간 가중치는 근거 없이 조정하지 마라

---

## 4. 매 스텝 state detach (초기 버전)

**기간**: epoch 1 ~ 60

**원인**: TBPTT를 구현한다고 매 타임스텝마다 `state.detach()` 호출

**왜 틀렸나**: 매 스텝 detach = BPTT 길이 1 = attention이 이전 스텝의 kappa를 참조할 수 없음 → attention 학습 원천 차단

**교훈**: detach의 의미를 정확히 이해해라. detach는 "여기서 gradient 흐름을 끊겠다"는 뜻이다. 매 스텝 끊으면 시퀀스 모델이 아니라 1-step 모델이 된다.

---

## 5. TBPTT 50스텝 기본값 설정

**기간**: epoch 21 ~ 174 (v2 전체)

**원인**: OOM 방지한다고 TBPTT를 50스텝으로 설정. "메모리 안전하게 하면서 attention도 학습되겠지"라는 근거 없는 판단.

**왜 틀렸나**: attention은 텍스트 전체에 대한 alignment를 학습해야 한다. 일반적인 단어가 100~300 스텝인데 gradient를 50스텝에서 끊으면, attention은 전체 alignment의 단편만 보게 된다. Alex Graves 원논문과 모든 레퍼런스 구현체는 **full BPTT**를 사용한다. 결과적으로 모델은 "길이 감각"만 얻고 "글자별 모양"은 전혀 학습하지 못했다.

**결과**: 174 에포크까지 돌렸지만 입력 텍스트와 무관한 필기체만 생성

**교훈**: 레퍼런스 구현체의 학습 방식을 먼저 확인해라. OOM이 걱정되면 batch_size를 줄여라, gradient 흐름을 끊지 마라.

---

## 6. 무조건 epoch 20부터 재시작 제안

**원인**: 코드를 수정할 때마다 "이전 학습은 무효"라고 판단하고 epoch 20부터 재시작을 권유

**왜 틀렸나**: LSTM/MDN 가중치는 이미 충분히 학습되어 있다. 문제는 attention뿐이고, full BPTT로 바꾸면 기존 체크포인트(epoch 170)에서 이어서 돌리면 attention만 새로 잡힌다. 학습된 가중치를 버릴 이유가 전혀 없다. 사용자에게는 24시간짜리 작업인데 "처음부터 다시"를 가볍게 말한 것은 리소스 감각이 없는 판단이다.

**교훈**: 코드를 고쳤다고 전체 학습을 버리지 마라. 어떤 가중치가 살릴 수 있고 어떤 게 무효인지 구분해라. 사용자의 시간과 리소스를 존중해라.

---

## 7. Loss를 timestep 수로 나눔 (total_loss / total_steps)

**기간**: 전 학습 기간 (epoch 1 ~ 243)

**원인**: 일반적인 시퀀스 모델에서 하듯이 "step당 평균 loss"를 계산. `total_loss / total_steps` 반환.

**왜 틀렸나**: 레퍼런스 구현체(X-rayLaser toolkit)는 timestep을 **합산**하고 batch_size로만 나눈다. 우리는 timestep으로도 나눠서 gradient가 1/T 스케일이 됐다. 평균 단어 길이 116스텝 기준으로 attention gradient가 레퍼런스 대비 **1/116**. LSTM/MDN은 직접 gradient 경로가 있어서 이 약한 신호로도 학습되지만, attention은 간접 경로(window → LSTM2/3 → MDN)라서 1/116 gradient로는 학습 불가.

**결과**: 243 에포크까지 돌려도 attention이 텍스트-stroke alignment을 못 잡음

**교훈**: 레퍼런스 구현체의 loss 정규화를 그대로 따라해라. "평균이 더 안정적이겠지"는 근거 없는 판단이다.

---

## 8. AdamW 옵티마이저 사용 + CosineAnnealing 스케줄러

**기간**: 전 학습 기간 (epoch 1 ~ 263)

**원인**: "Adam 계열이 범용적이니까" + "CosineAnnealing이 수렴에 좋으니까"라는 일반론으로 선택

**왜 틀렸나**: 레퍼런스 구현체는 `RMSprop(centered=True, alpha=0.95, eps=1e-4, momentum=0.9)` + **상수 lr=0.0001** (스케줄러 없음)을 사용한다. RNN+Attention 구조에서:
- Adam의 bias correction과 AdamW의 decoupled weight decay는 RNN gradient 스케일링을 왜곡
- CosineAnnealing은 lr을 0까지 떨어뜨려 학습 후반부에 attention gradient가 소멸
- `current_epoch`가 체크포인트에 저장 안 돼서 resume할 때마다 warmup 5회가 리셋됨

**교훈**: 옵티마이저와 스케줄러는 모델 아키텍처와 밀접하다. "범용적으로 좋은 옵티마이저"는 없다. 레퍼런스가 있으면 그대로 따라해라.

---

## 9. Gradient clipping 제거 + 방식 차이

**기간**: v4 이후 (gradient clipping 완전 제거)

**원인**: 레퍼런스를 보니 `clip_grad_norm_`을 안 쓰길래 우리도 제거

**왜 틀렸나**: 레퍼런스는 `clip_grad_norm_`이 아니라 `clip_grad_value_`를 쓴다. 이 둘은 완전히 다르다:
- `clip_grad_norm_`: gradient 벡터의 전체 L2 norm을 제한
- `clip_grad_value_`: 개별 gradient 값을 ±threshold로 자름
레퍼런스는 LSTM 파라미터 ±10, MDN 파라미터 ±100으로 **개별 값 클리핑**한다. 이걸 아예 안 하면 exploding gradient → 과적합 직행.

**교훈**: 같은 "gradient clipping"이라도 norm과 value는 완전히 다른 연산이다. 정확히 뭘 하는지 확인해라.

---

## 10. MDN sigma/rho에 불필요한 제약

**원인**: sigma에 `clamp(max=5.0)`, rho에 `* 0.95` 스케일링 적용

**왜 틀렸나**: 레퍼런스는 sigma에 아무 제약 없음 (순수 `exp()`), rho는 순수 `tanh()`. 제약을 걸면 모델이 표현할 수 있는 분포 범위가 줄어든다.

---

## 11. Teacher forcing에서 첫 스텝 누락

**원인**: `for t in range(T-1): input=strokes[t], target=strokes[t+1]`로 구현

**왜 틀렸나**: 레퍼런스는 `[zeros, strokes[0], ..., strokes[T-2]]`를 입력, `[strokes[0], ..., strokes[T-1]]`을 타겟으로 사용. 즉 **0벡터에서 시작**해서 첫 번째 스트로크도 예측한다. 우리 코드는 첫 스트로크를 건너뛰어서 inference 시 시작점 예측 능력이 없음.

---

## 12. Loss 평균 방식 차이 (per-step average vs sum/batch_size)

**원인**: 각 타임스텝에서 `(loss * mask).sum() / mask.sum()`으로 유효 샘플 평균을 구해 합산

**왜 틀렸나**: 레퍼런스는 모든 유효 타임스텝의 loss를 그냥 합산한 뒤 batch_size로만 나눈다. per-step average 방식은 짧은 시퀀스와 긴 시퀀스의 기여도를 왜곡한다.

---

## 13. PyTorch 표준 RMSprop 사용 (epsilon 위치 차이)

**기간**: 전 학습 기간

**원인**: `torch.optim.RMSprop`을 그대로 사용. "centered=True면 toolkit과 같겠지"라는 판단.

**왜 틀렸나**: toolkit의 CustomRMSprop은 `sqrt(variance + eps)` (epsilon이 sqrt 안), PyTorch는 `sqrt(variance) + eps` (epsilon이 sqrt 밖). variance가 0에 가까울 때 분모 차이가 100배. attention 파라미터처럼 gradient가 희소한 경우, PyTorch 버전은 100배 더 큰 업데이트를 가해 LSTM이 attention 없이 패턴을 외워버림.

**교훈**: "같은 알고리즘"이라도 구현 세부사항이 학습 결과를 완전히 바꿀 수 있다. 레퍼런스의 optimizer 코드를 줄 단위로 확인해라.

---

## 14. Dropout 제거

**원인**: toolkit에 dropout이 없길래 제거

**왜 틀렸나**: toolkit은 더 큰 데이터셋으로 학습한다. 우리 데이터는 ~8,000개로 hidden_size=400 3-layer LSTM에 비해 너무 작다. regularization 없이는 LSTM이 attention을 무시하고 학습 데이터 패턴을 직접 외운다. Train loss↓ Val loss↑가 epoch 1부터 시작.

**교훈**: 레퍼런스를 그대로 따라하되, 데이터 규모 차이는 반드시 고려해라. 작은 데이터에는 regularization이 필수다.

---

## 공통 패턴

> **증상을 보고 즉흥적으로 제약을 걸지 마라.**
> 음수 loss → clamp, 큰 sigma → clamp, 발산 → detach, loss 평균 → gradient 축소.
> 전부 원인 분석 없이 "안전하겠지"로 넣은 것이다.
> 
> **레퍼런스 구현체를 처음부터 정밀하게 비교했어야 했다.**
> 
> **옵티마이저, gradient clipping, loss 정규화는 모델 아키텍처의 일부다. 임의로 바꾸면 안 된다.**
